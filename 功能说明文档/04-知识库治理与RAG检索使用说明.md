# 知识库治理与 RAG 检索使用说明

知识库用于保存生成文章时可引用的真实资料。它是 RAG 检索的主要来源。

## 知识库适合保存什么

适合保存：

- 产品手册
- FAQ
- 安装说明
- 技术参数
- 故障排查
- 竞品分析
- 约束说明
- 风格参考

不建议保存：

- 真实客户案例。案例应进入 Case DB。
- 单个关键词。关键词应进入关键词库。
- 图片说明。图片应进入图片库。

## AI 自动分析如何填写知识库字段

知识库新增页和详情页都提供 AI 自动分析区。

粘贴资料后，AI 会尝试填入：

- 摘要 summary：重新整理后的资料概览，不应只是截取原文第一段。
- 描述 description：给管理员看的使用说明，说明这份资料适合什么生成场景、有什么边界。
- 正文 content：可直接入库的 Markdown 正文，应保留事实、表格、FAQ、步骤、限制条件和关键参数。
- 资料用途：自动推断对应的 knowledge_type、knowledge_role 和 importance。
- 关联 Entity 和标签建议：只匹配已有 Entity / 标签，不应新造无效标签。

### 表格和参数保真

如果资料中包含参数表、规格表、对比表或 FAQ 表格，AI 分析规则会优先保留：

- 型号
- 表头
- 单位
- 尺寸
- 电压
- 功率
- 速度
- 容量
- 温度
- 压力
- 百分比
- 测试条件

原则：

- 不合并相近型号。
- 不四舍五入。
- 不翻译产品型号。
- 不猜测缺失单元格。
- 表格清晰时转为 Markdown 表格。
- 表格不清晰时转为 key-value 或列表，并保留原始数值。

复杂表格仍建议人工复核。

## 补充分析要求

AI 自动分析区可以展开“补充分析要求”。

适合填写：

```text
请重点保留产品参数表、型号、单位、尺寸、电压、功率、速度、容量等数值，不要合并型号，不要四舍五入。
```

```text
请使用英文输出摘要、描述和结构化字段，品牌名、产品型号、单位和来源 URL 保持原样。
```

补充要求只作为本次分析的额外约束，不能覆盖系统的字段结构、事实准确和语言一致规则。

## knowledge_type 使用建议

knowledge_type 用于描述资料本身是什么。

推荐示例：

- product_manual：产品手册
- faq：常见问题
- troubleshooting：故障排查
- competitor_analysis：竞品分析
- technical_spec：技术参数
- installation_guide：安装指南
- style_reference：风格参考

注意：knowledge_type 不再使用 case_study，避免和 Case DB 重复。

## knowledge_role 使用建议

knowledge_role 用于描述生成文章时如何使用这份资料。

| role | 适用场景 | 生成时的意义 |
| --- | --- | --- |
| primary_source | 产品手册、官方参数、核心 FAQ | 作为事实依据优先引用 |
| supporting_context | 背景资料、补充说明 | 扩展内容深度 |
| constraint | 禁止事项、合规要求、技术限制 | 约束生成内容，避免乱写 |
| comparison_reference | 竞品分析、对比资料 | 用于比较和差异表达 |
| style_reference | 风格样稿、表达模板 | 参考写作风格，不作为事实来源 |
| archive | 旧资料、历史资料 | 默认不参与自动 RAG |

## 知识库状态

知识库状态会影响自动检索：

- active：正常参与检索。
- inactive：默认不参与自动检索。
- archive：默认不参与自动检索。

手动指定知识库时，系统仍允许引用 inactive 或 archive 资料，用于兼容旧任务或特殊需求。

## Entity 关联的重要性

知识库最好关联 Entity。

例如：

- SJ4060 产品手册关联 Entity：SJ4060
- 视觉灌胶机 FAQ 关联 Entity：视觉灌胶机
- 电池制造应用资料关联 Entity：电池制造

这样创建任务选择 Entity 后，RAG 可以优先找到真正相关的知识库，而不是只靠标签猜测。

## 多 Entity 独立关系

同一份知识库可以同时关联多个 Entity，并且每个 Entity 都可以单独设置关系。

例如一份 `SJ4060 产品手册` 可以这样关联：

| Entity | 关系 |
| --- | --- |
| SJ4060 | 主资料 / 主体知识 |
| 视觉灌胶机 | 辅助参考 |
| 电池制造 | 应用参考 |
| DJ771 | 竞品参考 |

后台交互方式：

1. 在知识库创建、编辑或详情页找到“与 Entity 实体的关系”。
2. 使用搜索多选框选择一个或多个 Entity。
3. 每个已选择 Entity 下方会出现一行 `Entity 名称 - 关系`。
4. 为每个 Entity 选择合适关系后保存。

Entity 编辑页的“关联知识库”使用同一套交互，只是方向相反：

- 在 Entity 页面选择多个知识库。
- 每个知识库后方单独设置它与当前 Entity 的关系。

默认关系会作为兜底值保留，旧数据和批量关联流程仍可正常使用。

## 批量治理

知识库列表支持批量操作：

- 批量分配 Collection。
- 批量追加标签。
- 批量设置 role。
- 批量关联 Entity。
- 批量归档或停用。

导入大量资料后，建议先完成这些治理动作，再进行批量生成。

## 重复与冲突治理检查

知识库列表页右上角提供“治理检查”入口。

它用于发现：

- 内容完全相同的知识库。
- 来源 URL 相同的知识库。
- 标题相同或标题/内容非常相似的知识库。
- 同来源、同 Entity 或同 Collection 相近资料中，电压、功率、尺寸、容量等数值型事实不一致的情况。

为避免误报，系统会先判断两份资料是否属于同一个规格主体：

- 如果两份资料分别关联到不同产品型号、产品线、产品/服务或材料主体，它们的 power、weight、voltage 等参数不同属于正常差异，不会提示为冲突。
- 如果两份资料只是共享品牌/公司 Entity，例如都属于 Robota，但产品不同，也不会仅因为共享品牌就提示规格冲突。
- 如果两份资料指向同一个产品型号或同一明确主体，参数不一致才会进入疑似冲突列表。

使用方式：

1. 进入“知识库列表”。
2. 点击“治理检查”。
3. 可选择 Collection 限定扫描范围。
4. 查看“疑似重复资料”和“疑似事实冲突”。
5. 点击对应知识库的“查看详情”进行人工复核，或点击“创建治理建议 / 创建审核建议”进入可追踪处理流程。

重要边界：

- 检查页本身只生成报告。
- 创建治理建议不会立即修改知识库内容。
- 重复资料建议需要进入详情页，并输入确认文本后才会应用。
- 应用重复资料建议时，只会把重复知识库状态改为 inactive，不会删除资料、删除切片或修改正文。
- 已应用的重复资料归档建议可以回滚，恢复处理前状态。
- 事实冲突建议只记录人工审核结论，不会自动覆盖知识库内容。
- 不会自动重建切片或 embedding。
- “疑似冲突”只是提示，需要结合来源可靠度人工判断。
- 自动合并正文仍然禁用，除非后续单独设计更严格的合并、版本和回滚策略。

## 切片与向量化任务状态

知识库保存后，系统会把切片与向量化放入后台队列处理，避免大文件长时间阻塞页面。

可见状态：

- 未生成切片：还没有任何 knowledge chunks。
- 已有切片：旧数据或已存在切片，但没有新队列状态。
- 已排队：任务已进入队列，等待 worker 执行。
- 处理中：worker 正在重建切片并写入向量。
- 已完成：切片任务完成。
- 失败：任务执行失败，可查看错误信息并重试。

在哪里看：

- 知识库列表页：每条知识库会显示切片任务状态和向量化数量。
- 知识库详情页：在标签区域下方可以看到“切片任务状态”卡片。

使用建议：

- 大文件导入后看到“已排队”或“处理中”是正常现象，不需要重复点击。
- 如果状态长期停留在“已排队”，通常表示队列 worker 没有运行。
- 如果状态为“失败”，先检查错误信息、Embedding 模型配置和 API 限流，再点击“重试切片”。
- 旧知识库可能显示“已有切片”，这表示它已经有 chunks，但不是这套新状态流程生成的记录。

性能补强阶段已确认该队列流程具备状态写回、失败记录和重复投递保护。后续如果导入超大文件，重点检查 worker 是否正常运行、Embedding 模型是否限流，以及失败后是否能从页面重试。

## RAG 检索流程

生成文章时，系统会综合使用：

1. 当前任务 Collection。
2. 任务选择的 Entity。
3. Entity 关联知识库。
4. Entity 关联 Case。
5. 任务选择的知识库。
6. 受控标签筛选。
7. 知识库状态和 role。
8. 知识库与 Entity 的关系类型。

最终文章会记录生成来源和检索 chunk，便于审核。

## RAG 解释怎么看

在文章编辑页的“生成来源”卡片中，可以查看本次生成的知识检索解释。

重点看这些信息：

- 策略：本次主要使用 pgvector、混合检索或其他召回方式。
- 平均证据分：越高通常代表片段和当前文章主题越贴近。
- 使用知识库：确认引用的资料是否来自正确 Collection、Entity 或手动选择的知识库。
- 召回来源：判断片段来自向量检索、关键词匹配、Entity 关联还是人工指定。
- 匹配原因：用于解释为什么这个片段被选中，例如语义相似、资料优先级、Entity 关联或关键词命中。
- 评分组成：用于排查“为什么某个片段分高或分低”，例如向量、词面、元数据和距离等因素。

如果发现文章内容不准，建议先检查 RAG 解释里的“使用知识库”和“匹配原因”，再决定是补充知识库、调整 Entity 关联，还是降低某些资料的使用优先级。

## AI 知识库纠错助手

当发现知识库资料本身可能有错误，或文章生成内容暴露出知识来源问题时，可以使用 AI 知识库纠错助手。

入口：

- 知识库详情页：适合直接针对某一份知识库资料提出错误描述。
- 文章编辑页：适合选中文章里的错误段落，再让系统反查可能相关的知识片段。
- 纠错记录页：`/admin/knowledge-corrections`，用于集中查看待审核、已批准、已拒绝和已应用的纠错记录。

使用流程：

1. 输入错误描述，例如“该型号的工作行程应该是 300*300mm，不是 400*400mm”。
2. 可选：在文章编辑页先选中错误段落，再点击填入选中文本。
3. 选择 AI 模型，或保持自动选择。
4. 提交后，AI 会检索相关 knowledge chunks，并生成纠错提案。
5. 管理员进入纠错详情页，查看原文、建议内容、错误类型、理由和置信度。
6. 确认无误后先批准，再点击应用。
7. 系统会更新对应知识片段和知识库正文中的原片段，并刷新该片段 embedding / fallback vector。

重要边界：

- AI 不会直接覆盖知识库内容。
- 被拒绝或已经应用的纠错不能再次应用。
- 如果原知识片段在审核期间已经被其他操作修改，系统会阻止应用，避免覆盖新内容。
- 每次应用都会保存版本记录，可从纠错详情页回滚到旧内容。
- 当前页面采用“原文 / 建议内容”并排对比；字符级 diff 可作为后续增强。

建议用法：

- 文章生成后发现事实错误时，优先从文章编辑页发起纠错。
- 明确知道某份知识库有错误时，优先从知识库详情页发起纠错。
- 如果只是文章表达不够好，但知识库事实正确，不建议使用纠错助手，应直接编辑文章或优化生成提示词。
