# Collection 业务容器使用说明

Collection 是 GEOFlow 的顶层业务容器，用来划分不同业务线或内容资产边界。

## 适合放入 Collection 的内容

Collection 应该表示稳定、长期存在、需要独立管理的一类业务：

- 自动化设备
- 工业制冷
- 色选设备
- 照明

## 不适合放入 Collection 的内容

以下内容不建议作为 Collection：

- 单个产品型号，例如 SJ4060。
- 单篇文章主题。
- 临时活动。
- 具体素材类型，例如 FAQ、产品手册。
- 普通属性，例如高精度、节能、英文内容。

这些内容更适合使用 Entity、知识库 type、知识库 role 或标签表达。

## 在系统中的入口

Collection 入口保留在后台顶部菜单中。

基础素材库模块中不再展示业务容器卡片，避免它和普通素材库混在一起。

## Collection 健康度

Collection 列表页会显示每个 Collection 的健康度分数，并提供“健康度”入口。

健康度用于判断这个业务容器是否已经具备生成文章的基本素材条件。

检查内容包括：

- 是否有 Entity。
- 是否有知识库。
- 是否有标题库。
- 是否有图片库。
- 是否有 Case。
- 知识库是否关联 Entity。
- Case 是否关联 Entity。
- 知识库切片是否已向量化。
- 标签是否使用受控白名单分组。
- 是否存在重复标签。

健康度只是只读体检，不会自动修改、删除、合并或重建素材。

建议在以下场景查看健康度：

- 新建 Collection 后，准备正式创建生成任务前。
- URL 采集导入大量资料后，检查是否缺少 Entity 或向量化。
- 某个 Collection 生成文章质量不稳定时，先排查素材结构是否完整。

## 创建任务时的作用

创建任务页中 Collection 为必选项。

选择 Collection 后，后续素材筛选会以该业务容器为基础范围，包括：

- Entity
- Case
- 知识库
- 关键词
- 图片
- 标签筛选

## 推荐规则

1. 一个素材只能归属于最合适的 Collection。
2. 跨业务复用资料时，优先拆分资料或创建通用 Collection，不建议靠大量标签硬凑。
3. Collection 不要频繁改名，避免团队理解混乱。
4. 如果 Collection 太多，说明业务容器设计过细，需要合并。

## 示例

如果有产品 SJ4060，属于自动化设备业务：

- Collection：自动化设备
- Entity：SJ4060
- 产品线标签：视觉灌胶机
- 产品型号标签：SJ4060
- Topic 标签：参数、应用场景、FAQ
- 知识库：SJ4060 产品手册、SJ4060 FAQ、SJ4060 故障排查
