# 轻量 CRM 与单据制作使用说明

本文档说明 GEOFlow 新增的轻量 CRM 工作流。该模块不是完整 ERP，不处理财务总账、库存核算、采购或生产排程；它用于把客户、询盘、知识资料和报价初稿连接起来。

## 入口

后台顶部菜单新增 `CRM`。

CRM 下目前包含：

- 工作台
- 客户
- 询盘
- 商机
- 待办
- 单据制作
- 订单
- 售后
- 内容候选

## 客户管理

客户是 CRM 的基础对象。每个客户可关联一个 Collection（业务容器），后续询盘和报价会沿用该容器。

### 基础字段

- **公司名**：客户公司的名称，作为客户列表和详情页主标识。
- **主联系人**：最常用的外部联系人；创建客户时填写的联系人会自动成为主联系人。

### 选填字段

- **公司名**：客户所在的公司名称，补充信息。在单据打印中会进入买方信息的 Company 栏。
- 业务容器 Collection
- 客户类型
- 行业
- 国家
- 地址
- 来源渠道
- 联系电话
- 邮箱
- 职位
- 官网
- 负责人（内部员工）
- 状态（活跃 / 潜在 / 不活跃 / 暂停合作）
- 备注

客户类型和来源渠道使用受控选项，减少后续筛选和统计时的命名混乱。国家字段提供可搜索候选，仍保留历史值兼容。

### 多联系人

同一家公司可以维护多个外部联系人，并指定一个主联系人。主联系人姓名、职位、电话和邮箱会继续同步到旧客户字段，供现有单据“从客户资料带入”功能使用。

这里的联系人指客户公司的人员，不是 GEOFlow 内部员工。内部负责员工仍由“负责人”字段表达。

### 详情页功能

客户详情页现在是单个客户的 CRM 总览页。顶部会显示该客户的询盘、活动中商机、单据金额、订单、待处理售后和未完成待办数量，方便先判断这个客户目前处在哪个业务阶段。

客户详情页可维护和查看：

- 基础信息（含公司名和联系人划分）
- 销售链条：相关询盘、商机、单据、订单和售后工单
- 活动记录（已发生的电话、邮件、会议和沟通结果）
- 客户待办（未来要完成的动作和截止时间）
- 该客户相关询盘列表
- 该客户相关报价列表

询盘活动按询盘独立展示，不再混入同一客户其他询盘的记录。客户详情仍可汇总查看该客户的全部活动。

### 归档客户

客户、询盘、单据、订单和售后使用归档方式移出默认列表。归档客户不会删除其询盘、单据、订单或售后工单，商业记录中的客户关系会继续保留。

## 询盘管理

询盘用于记录客户需求，并把需求连接到现有素材系统。

询盘可关联：

- Collection
- 客户
- 负责人（内部员工）
- Entity
- 知识库
- Case
- 标签

选择 Collection 后，Entity、知识库和 Case 推荐选项会限制在当前 Collection 内。

### 询盘状态

询盘状态只表达需求处理状态，不再承担报价、赢单、输单等销售管道职责。

新建或编辑询盘时建议使用：

- 新询盘
- 分析中
- 已确认
- 已转商机
- 无效
- 已关闭

历史数据中的“已报价 / 赢单 / 丢单”仍可兼容显示，但后续销售推进建议进入商机模块维护。

### 详情页功能

询盘详情页展示：

- 需求识别结果（客户、负责人、语言、需求摘要、产品兴趣、回复要点）
- 当前询盘的活动记录
- 当前询盘的未来待办
- “转为商机”入口：确认存在真实采购可能后，一键创建商机
- 关联商机列表：已有商机时显示商机名称、阶段、金额和跳转入口
- 报价记录和订单记录：保留旧流程可追溯，但销售推进建议进入商机

## 商机管理

询盘只有在确认存在真实采购可能后，才建议转为商机。普通咨询、样本索取或信息不完整的线索可以继续留在询盘中。

商机管道阶段为：已确认、需求梳理、方案制定、报价方案、商务谈判、赢单、输单。每个商机可维护金额、币种、成交概率、预计成交日期、竞争对手、输单原因和内部备注；进入输单阶段时必须填写输单原因。

商机不再维护独立的“下一步动作”字段。未来要做的事情统一进入 `CRM 待办`，这样客户页、询盘页、商机页和工作台看到的是同一条待办。

### 商机看板

CRM 导航中提供“商机看板”入口。看板按 7 个商机阶段展示当前商机卡片，适合快速判断销售机会集中在哪些阶段。

每张商机卡片会显示：

- 商机名称
- 客户名称
- 金额和成交概率
- 预计成交日期
- 来源询盘
- 当前下一步待办

点击卡片会进入原商机编辑页。当前看板是只读/轻交互视图，不支持拖拽修改阶段；需要调整阶段时，进入商机编辑页保存，避免误拖动导致销售状态被改错。

### 销售链路 V2 规则

当前 CRM 已完成销售链路 V2 收口：

- 商机必须明确来源：从询盘创建，或标记为无来源直接创建。
- 同一询盘只能有一个活动商机；需要测试或重复记录时，请先归档旧商机。
- 从询盘转商机时，系统会原位关联未完成待办和未归属单据，不复制记录。
- 客户、询盘、商机、单据和 Collection 必须保持同一条销售链，冲突组合会拒绝保存。
- 活动记录表示已经发生的沟通；CRM 待办表示未来动作。活动可以顺手创建待办，待办完成时可以写回活动结果。
- 商机归档不会删除关联待办、活动和单据；恢复时如果来源询盘已被其他活动商机占用，会阻止恢复。

系统提供 `crm:pipeline-audit` 审计命令。默认只读；`--apply` 只修复唯一候选的历史断链。没有唯一候选的旧数据需要人工判断，不会由系统自动猜测。

### 从询盘转为商机

在询盘详情页点击“转为商机”后，系统会自动带入：

- Collection
- 客户
- 来源询盘
- 商机名称（默认使用询盘标题）
- 负责人（能匹配到后台管理员时自动带入）
- 需求摘要、产品兴趣、建议回复要点等备注信息

创建成功后，来源询盘状态会自动变为“已转商机”。如果询盘已经存在关联商机，页面会显示“查看商机”，避免重复创建。

转换时系统会在确认框中展示影响摘要：将补关联多少个未完成待办、多少份已有单据。转换不会复制记录，只会把这些原有记录补上商机关系。

系统现在遵循“一条询盘最多一个活动商机”的规则：普通入口和数据库约束都会阻止同一询盘重复创建活动商机。确实不来自询盘的销售机会，可以在商机新增页选择“无来源直接创建”。

### 商机详情页

商机编辑页右侧会显示来源询盘上下文，包括：

- 来源询盘标题
- 需求摘要
- 仍需确认的问题
- 关联 Entity、知识库、Case 数量
- 返回询盘详情的入口

商机页也提供“新建单据”入口和关联单据列表，适合从销售机会继续生成报价单、PI、合同等文档。

商机可以归档和恢复。归档只会把商机移出活动管道，不会删除关联待办、活动记录或单据。恢复时如果来源询盘已经被其他活动商机占用，系统会拒绝恢复，避免重复商机。

### 单据链路视图

询盘详情、商机编辑页和客户详情页都提供单据链路摘要。它用于快速确认一条业务是否已经从客户需求推进到单据、订单或售后阶段。

链路视图会按顺序展示：

- 客户
- 询盘
- 商机
- 单据（报价单、PI、发票、装箱单、合同等）
- 订单
- 售后工单

每个节点都可以点击进入对应详情页。这个视图只做关系总览，不替代原有的详细列表和编辑入口，也不会自动创建副本或改变单据版本。

推荐用法：

- 在询盘页确认“这个询盘是否已经转为商机、是否已经出过单据”。
- 在商机页确认“当前商机下的单据是否已经形成订单或售后工单”。
- 在客户页快速查看“这个客户整体业务链路走到哪一步”，再进入下面的销售链条详细列表处理具体记录。

## 活动与待办

- **活动记录**：已经发生的事情，例如收到邮件、完成电话、举行会议、确认技术需求。
- **CRM 待办**：未来需要完成的事情，例如明天下午回访、周五发送报价、下周确认样品结果。

记录活动时需要选择一个结构化活动类型：

- 普通记录
- 电话沟通
- 邮件往来
- 会议沟通
- 在线聊天
- 单据沟通
- 待办完成
- 系统记录

原来的“活动类型”自由输入仍保留为补充备注，用于记录更细的说明，例如“WhatsApp”“客户内部会议后反馈”“PI 付款条款确认”等。历史活动没有结构化类型时，会默认显示为普通记录。

活动记录输入框使用轻量 Markdown 编辑器，支持 `编辑 / 预览 / 源码` 视图，以及标题、加粗、引用、列表、链接等快捷按钮。它适合记录沟通结论、客户原话、确认事项和后续背景，不替代 CRM 待办。

客户、询盘和商机现在共享同一条活动事实源。商机详情页也可以直接记录活动；如果商机来自询盘，这条活动会同时显示在客户页、询盘页和商机页。

记录活动时可以勾选“同时创建下一步待办”。系统会自动把新待办关联到同一个客户、询盘和商机。

CRM 工作台会集中显示当前管理员的今日和逾期待办。完成待办时可以直接完成，也可以填写完成结果；填写结果后系统会自动生成一条“待办完成”类型的活动记录。需要继续处理时可重新打开待办。

## AI 需求识别

询盘创建/编辑页提供 `AI 需求识别` 区域。

使用方式：

1. 粘贴客户邮件、聊天记录或询盘正文。
2. 可选择 AI 模型，也可以保留自动选择。
3. 点击分析。
4. 系统会填入需求摘要、产品兴趣、回复要点、缺失问题，并推荐已有 Entity、知识库和 Case。
5. 管理员可以手动增删推荐结果后保存。

注意：

- AI 不会直接创建新 Entity、知识库、Case 或标签。
- AI 只推荐当前已有对象。
- 没有可用 AI 模型时，系统会使用本地关键词匹配作为降级推荐。

## 单据制作（报价管理）

单据（Quotes）是报价和文档生成的核心入口，覆盖从报价到出货的完整单据链。

### 创建方式

- 从客户直接创建新单据
- 从询盘创建：自动带入客户、负责人、Collection；若询盘关联了 Entity，明细会预填 Entity 名称
- 从商机创建：自动带入商机、客户、Collection，并保留来源询盘关系
- 从单据列表页的"新增单据"按钮创建

### 基础信息

- **客户**：必选，选择后可使用"从客户资料带入"自动填充买方信息
- **业务容器**：可选，限制后续 Entity 和图库候选范围
- **负责人**：内部员工
- **关联询盘**：可选，用于追溯客户原始需求
- **关联商机**：可选，用于把报价、PI、合同等单据挂到销售管道
- **单据号**：自动生成，可手动修改
- **文档类型**：报价单 / 形式发票 / 正式发票 / 装箱单 / 合同
- **标题**：必填
- **币种**：默认 USD

### 单据链一致性

单据保存前会校验客户、询盘、商机和 Collection 是否属于同一条销售链：

- 只选择询盘时，如果该询盘已有活动商机，系统会自动补上商机。
- 只选择商机时，如果商机来自询盘，系统会自动补上来源询盘。
- 如果询盘和商机不是同一条链、客户不一致或 Collection 冲突，系统会拒绝保存。
- 从商机创建单据时，会自动带入商机、客户、Collection 和来源询盘。

这个规则可以避免同一份报价挂到错误客户或错误销售机会下。

### 买方信息

买方信息模块位于基础信息下方，所有字段支持手动编辑。右上角"从客户资料带入"按钮可一键将客户资料填入对应字段：

| 表单标签 | 字段名 | 自动带入来源 |
|---------|--------|-------------|
| 联系人 | `buyer_contact` | 客户的"客户名称（联系人）" |
| 公司名 | `buyer_company` | 客户的"公司名" |
| 电话 | `buyer_phone` | 客户的联系电话 |
| 邮箱 | `buyer_email` | 客户的邮箱 |
| 国家 | `buyer_country` | 客户的国家 |
| 地址 | `buyer_address` | 客户的地址 |

买方信息在打印单据的 Buyer/Importer（或 Consignee/Buyer）面板中展示。联系人优先显示，公司名作为补充（格式：`联系人 · 公司名`）。

### 商业信息

- **Trade Term**：FOB / CIF / EXW 等贸易条款
- **Origin**：原产国
- **装运港 / 目的港**：物流信息
- **运输方式**：Sea Freight / Air Freight 等
- **唛头（Shipping Mark）**：可选，用于装箱单
- **有效期**：报价有效期
- **交期**：预计交付时间
- **质保条款**
- **安装条款**
- **包装条款（Packing）**：默认值 "Standard export wooden case"，可修改

### Bank Account JSON（卖方银行信息）

仅在形式发票打印时展示。首次为空时预填模板 JSON，可手动编辑。标准字段：

- Beneficiary（受益人）
- Bank Name（银行名称）
- Account No.（账号）
- Bank Code（银行代码）
- Branch Code（分行代码）
- SWIFT
- Currency（币种）
- Bank Address（银行地址）

### Seller Company JSON（卖方公司信息）

用于覆盖打印页的卖方信息。首次为空时预填模板 JSON。字段：

- name（公司名）
- address（地址）
- phone（电话）
- email（邮箱）
- website（网站）

打印时优先级：单据级 `Seller Company JSON` > 站点设置默认值。

### 明细行

每条明细包含：

- **行类型**：product / accessory / service / spare_part / training / other
- **Entity 关联**：选择已有 Entity 可自动预填型号和描述
- **型号**：产品型号
- **HS Code**：海关编码
- **产品图片**：从图片库选择或本地上传（单张不超过 200KB）
- **名称 / 描述**
- **数量 / 单位**
- **单价**
- **包装件数 / 净重 / 毛重 / 体积**：物流字段，报价单和形式发票阶段可留空
- **包装尺寸 L×W×H (cm)**：装箱单专用

明细小计 = 数量 × 单价，空白行不保存。支持动态增删行。

### 单据类型联动

选择不同单据类型时，明细中的部分字段自动显示/隐藏：

- **报价单 / 形式发票**：显示价格、有效期、银行信息（形式发票）；物流字段可折叠
- **正式发票**：显示价格和 HS Code，顶部含 Exporter/Seller + Importer/Buyer 双面板
- **装箱单**：隐藏价格，显示包装尺寸、重量、体积，顶部含 Shipper/Seller + Consignee/Buyer 双面板，含 Shipment / Shipping Mark / Port Info 三面板
- **合同**：显示价格和自定义合同条款，右侧含 Seller Info 面板

### 金额计算

- 明细小计 = Σ(数量 × 单价)
- 最终合计 = 明细小计 + 运费 + 税费 - 折扣

### 打印页

打印页按文档类型分流渲染，各类单据标题和面板命名如下：

| 单据类型 | 卖方面板 | 买方面板 |
|---------|---------|---------|
| 报价单（Quotation） | - | Buyer / Customer |
| 形式发票（PI） | - | Buyer / Customer |
| 正式发票（CI） | Exporter / Seller | Importer / Buyer |
| 装箱单（PL） | Shipper / Seller | Consignee / Buyer |
| 合同（Contract） | Seller Info | Buyer/Importer |

所有打印页卖方联系人（Contact）映射到单据负责人（owner）。卖方公司名、地址、电话、邮箱和网站来自站点设置或单据级 `Seller Company JSON`。

每条单据的打印页可通过单据列表的"打印"入口打开，也可在详情页通过"打印单据..."下拉直接选择目标类型。单据详情页和打印预览页都提供"下载 PDF"入口，系统会使用 Chromium/Puppeteer 按当前单据类型生成 A4 PDF。明细较多时，系统会按单据类型、图片、长描述和装箱字段自动拆分第一页、续页和末页，并生成动态页码。

如果 PDF 生成失败，页面会提示先使用打印预览手动保存为 PDF。HTML 打印预览仍然是兜底方案，不影响单据编辑和打印。

### PDF 视觉回归检查

每次修改单据打印模板或 Docker PDF 环境后，可在项目目录执行：

```bash
KEY=$(docker exec geoflow-app grep '^APP_KEY=' /var/www/html/.env | cut -d= -f2-)
docker exec -e APP_KEY="$KEY" geoflow-app php artisan crm:document-pdf-regression
```

该命令会自动选择当前数据库中的真实单据样本，生成报价单、形式发票、正式发票、装箱单和合同五类 PDF，并输出 HTML、PDF、PNG 截图和 `report.md`。输出目录位于 `storage/app/pdf-regression/日期时间/`。

### 单据类型输出

同一份单据可以临时按报价单、形式发票、正式发票、装箱单或合同格式打印。打印类型切换只影响本次预览和输出，不会新增数据库记录，也不会把单据复制成多个副本。

以前的“创建副本”入口已从前端移除。原因是为同一个询盘或商机创建多份独立副本会让单据列表快速变乱。后续如果需要正式区分报价单、PI、CI、装箱单和合同的独立版本，应先做“单据包 / 版本管理 / 派生单据折叠展示”，再恢复独立副本能力。

## 订单管理

订单用于轻量跟踪报价被接受后的执行状态。

### 创建方式

- 单据详情页点击"生成订单"
- 单据列表页操作列点击"创建订单"

系统会复制报价的客户、Collection、明细、币种和总金额。在订单详情或编辑页维护付款、生产、交付和订单状态。

订单管理只做状态跟踪，不处理库存、财务、采购或生产排程。

## 售后工单

售后工单用于记录真实问题，并把问题连接到现有知识体系。

工单可关联：

- Collection
- 客户
- 订单
- 核心 Entity
- 多个知识库
- 多个 Case

工单创建/编辑页提供 `AI 工单分析`，使用方式与询盘 AI 需求识别类似。

## 内容候选草稿

内容候选用于把 CRM 中的真实需求转成可入库素材，但不会直接写入素材库。

- 从询盘生成标题候选 / FAQ 候选
- 从售后工单生成 FAQ 候选 / Case 候选
- 在 `CRM > 内容候选` 中审核，确认应用后写入对应素材库
- 拒绝的候选不会写入任何素材库

## 创建任务时引用 CRM 来源

创建任务页 `CRM 来源` 区域可选来源：客户、询盘、售后工单。用于记录文章来源、方便后续回溯。

## 推荐工作流

1. 创建或选择 Collection
2. 创建客户，填写公司名和主联系人；需要时补充其他联系人
3. 创建询盘，选择客户、Collection 和负责人
4. 使用 AI 需求识别，关联 Entity、知识库和 Case
5. 确认存在采购可能后转为商机，设置下一步待办
6. 从询盘或商机生成单据，完善买方信息、商业信息和明细；系统会自动校验单据链
7. 需要 PI、CI、PL 或合同时，在详情页或打印页切换打印类型输出，不创建额外副本
8. 报价被接受后，从单据生成订单
9. 订单执行或售后过程中创建售后工单，并把高频问题沉淀为内容候选
10. 创建文章任务时，可选择相关询盘、客户或售后工单作为 CRM 来源
