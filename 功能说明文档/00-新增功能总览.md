# GEOFlow 新增功能使用总览

本文档汇总从本轮定制开发以来新增的主要功能，以及它们在实际内容生产中的职责边界。

## 核心思路

GEOFlow 现在不再只是“选择素材生成文章”，而是升级为：

1. 先用 Collection 划分业务边界。
2. 再用 Entity 建立知识索引节点。
3. 用知识库保存真实资料。
4. 用 Case 保存真实案例。
5. 用标签描述可复用属性。
6. 创建任务时通过 Collection、Entity、Case、标签和知识库共同限定生成上下文。
7. 文章生成后自动记录来源、RAG 引用和质量评分，方便审核。
8. 用轻量 CRM 把客户、询盘、报价、订单和售后问题连接到素材系统，并将真实业务问题沉淀为内容候选。
9. 用 Markdown 编辑器完成文章二次编辑，并按需要复制到公众号/微信等富文本编辑器。
10. 用模板工厂在隔离草稿中复刻参考站点风格，再由管理员确认是否发布为站点主题。
11. 用 Codex 业务录入助手 API 把自然语言业务想法整理成待审核草稿，管理员确认后再写入 CRM 或内容候选。

## 新增模块职责

| 模块 | 作用 | 不建议承担的职责 |
| --- | --- | --- |
| Collection | 顶层业务容器，例如自动化设备、工业制冷、色选、照明 | 不要用普通标签替代 Collection |
| Entity | 知识索引节点，例如 SJ4060、桌面点胶机、PU 树脂、电池制造 | 不要把所有长文资料塞进 Entity 字段 |
| Knowledge Base | 保存产品手册、FAQ、竞品资料、故障排查等真实资料 | 不再存放案例型内容 |
| Case DB | 保存客户案例、应用场景、项目结果 | 不作为普通知识库 role |
| Tag | 描述轻量可复用属性，例如 Topic、Audience、Intent | 不创建无限分组，不承担 Collection、Entity 或知识库资料类型职责 |
| Article Quality | 判断文章是否完整、是否引用知识库、是否覆盖关键词、是否存在审核风险 | 不自动替代人工最终审核 |
| Knowledge Correction | 发现知识库事实错误后，由 AI 生成纠错提案，管理员审核后应用并保留版本 | 不让 AI 自动覆盖知识库，不替代人工确认 |
| Lightweight CRM | 管理客户、询盘、报价、订单、售后工单，并生成待审核的标题、FAQ、Case 候选 | 不承担完整 ERP 的库存、采购、生产排程或财务结算 |
| Article Editor | 使用 Markdown 编辑器编辑文章、上传编辑器图片、复制 Markdown 或公众号/微信格式 HTML | 不替代文章事实审核和质量检查 |
| Template Factory | 从首页、列表页、文章页 3 个参考页面生成主题草稿、预览、迭代、发布或打包 | 不直接覆盖现有主题，不绕过预览和管理员确认 |
| Codex Intake Assistant | 通过 API 把 Codex / AI 整理出的客户、询盘、售后、FAQ、Case 等建议保存为待审核草稿 | 不让 Codex 直接写最终业务表，不跳过管理员审核 |

## 推荐使用顺序

1. 创建 Collection。
2. 创建或导入 Entity。
3. 导入知识库，并选择“资料用途”、Collection 和关联 Entity。
4. 创建 Case，并关联最直接的 Entity。
5. 给关键词、图片、知识库、Entity、Case 选择受控标签。
6. 在创建任务页选择 Collection。
7. 根据文章主题选择相关 Entity 和 Case。
8. 按需使用受控分组标签进一步筛选素材。
9. 生成文章后查看“生成来源”和“质量检查”。
10. 对低分文章优先人工审核。
11. 如果发现事实错误，先确认是文章表达问题还是知识库源资料问题；源资料错误时使用 AI 知识库纠错助手生成提案并人工应用。
12. 把询盘和售后工单中的高频问题转成内容候选，人工确认后再写入标题库、知识库或 Case DB。
13. 在文章编辑页用 Markdown 编辑器完成二次编辑，必要时复制公众号/微信格式 HTML 到外部平台。
14. 需要新增前台视觉风格时，在网站设置的模板工厂中先创建草稿、预览和迭代，再决定是否发布。
15. 如果在 Codex 中整理客户沟通或售后问题，先通过 Assistant context search 查上下文，再创建 AI 录入草稿，最后到后台草稿箱确认应用。

## 关键变化

- Collection 是必选业务容器，标签不再承担顶层分类职责。
- Entity 不是资料正文仓库，而是把知识库、图片、关键词、Case 串起来的索引节点。
- 知识库可以关联多个 Entity，并且每个 Entity 都可以单独设置关系，例如主资料、辅助参考、应用参考或竞品参考。
- Case 类型已经改为受控选项，生成时会根据案例类型决定引用方式。
- 知识库 role 会影响 RAG 检索和生成时的使用方式。
- URL 智能采集生成的 Case 会归一到受控 Case 类型，避免继续产生自由类型。
- URL 智能采集创建任务时可以选择 AI 分析模型；不指定时按模型优先级自动选择。
- 后台 AI 模型配置页可为聊天模型单独设置“正文最大输出 Token”，用于降低长文生成被模型默认上限截断的概率；留空时使用系统默认值。
- Knowledge、Entity、Case 的 AI 自动分析已统一增强提示词规则，重点约束语言一致、事实准确和表格参数保真。
- AI 自动分析区提供“补充分析要求”，用于临时强调表格、语言或案例提取策略，但不会覆盖系统字段规则。
- AI 知识库纠错助手已接入知识库详情页和文章编辑页；AI 只生成 proposal，管理员应用后才会更新知识片段并刷新 embedding。
- 标题库可以保留库级标签；关键词库和图片库不再使用库级标签，避免误以为库内素材都继承该标签。
- 标签分组已经改为白名单治理，可在标签管理页维护允许使用的分组。
- 关键词库、标题库、图片库和知识库删除后，会保留当前 Collection / 筛选视图并回到列表位置。
- 轻量 CRM 已支持客户、询盘、报价、订单、售后工单、内容候选和创建任务页 CRM 来源记录。
- 文章创建/编辑页已接入 Vditor Markdown 编辑器，可复制 Markdown 或公众号/微信格式 HTML。
- 编辑器上传图片会写入图片库并在有文章 ID 时关联当前文章。
- 网站设置页新增模板工厂入口，模板复刻先生成隔离草稿，不直接覆盖现有主题。
- Codex 业务录入助手 API 已支持只读上下文搜索、草稿创建、草稿预检、后台审核应用和基础治理提醒。
- Codex 创建的知识库 / Case 动作只进入 CRM 内容候选，不会直接覆盖知识库正文或直接写入 Case DB。
- 分发首页最近日志已改为分页显示，避免日志过多时撑长页面。
- 首次部署时登录页可以显示默认管理员账号提示；生产环境可关闭该提示。
