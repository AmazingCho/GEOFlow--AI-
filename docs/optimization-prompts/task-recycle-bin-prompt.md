# 后续优化提示词：任务回收站与文章保留

> 状态：已实现归档提示词，仅作历史参考。任务回收站核心功能已完成：任务删除为软删除，文章和来源任务关系保留，回收站支持查看和恢复。后续不要把本文当作待执行计划；如需永久删除任务，必须先重新确认数据保留策略。

请在 GEOFlow 中实现“任务回收站”功能，要求保护现有文章数据，不破坏当前任务创建、编辑、运行、暂停、删除、文章生成和分发流程。

## 背景

当前任务删除后，通过该任务生成的文章会被同步删除或解除关联，导致历史文章和生成来源不易追踪。现在希望删除任务时保留文章，并将任务放入回收站，管理员可以查看、恢复或永久删除任务。

## 核心目标

1. 删除任务时，不删除由该任务生成的文章。
2. 文章列表和文章编辑页仍能显示来源任务信息。
3. 如果任务已被删除，文章来源显示为“已删除任务”，并尽量显示任务原名称。
4. 被删除任务进入任务回收站。
5. 管理员可以在回收站中恢复任务或永久删除任务。
6. 不破坏现有任务运行、暂停、生成、分发、监控和文章管理流程。

## 数据库设计

请优先采用 Soft Deletes，而不是立即物理删除任务。

### tasks 表

新增或确认以下字段：

- `deleted_at`：软删除时间。
- 如当前任务表已存在 `deleted_at`，复用即可。

### articles 表

为保留任务来源快照，建议新增：

- `source_task_id`：原任务 ID，可为空。
- `source_task_name`：任务名称快照。
- `source_task_deleted_at`：任务删除时间快照，可为空。

说明：

- 如果当前文章已有 `task_id`，不要直接移除。
- 删除任务时，可以保留 `task_id` 指向软删除任务。
- 文章展示时应支持 `withTrashed` 查询任务。
- 如果永久删除任务，文章仍通过 `source_task_name` 显示历史来源。

## 后端逻辑

### 删除任务

修改 `TaskLifecycleService::deleteTask` 或现有删除入口：

1. 不删除文章。
2. 不软删除文章。
3. 写入文章的来源任务快照：
   - `source_task_id`
   - `source_task_name`
   - `source_task_deleted_at`
4. 停止任务相关调度或运行状态。
5. 将任务软删除。
6. 写入 admin activity log。

### 恢复任务

新增恢复接口：

- 恢复软删除任务。
- 恢复后任务状态建议为 `paused`，避免恢复后立即自动运行。
- 文章来源显示恢复为正常任务链接。
- 写入 admin activity log。

### 永久删除任务

新增永久删除接口：

- 只允许删除已在回收站中的任务。
- 永久删除任务时，不删除文章。
- 文章继续显示 `source_task_name`，状态显示“已删除任务”。
- 写入 admin activity log。

## 前端 UI

### 任务管理页

1. 增加“回收站”入口。
2. 删除按钮文案改为更明确的“移入回收站”。
3. 删除确认弹窗说明：
   - 任务会停止并移入回收站。
   - 已生成文章会保留。
   - 可在回收站恢复任务。

### 任务回收站页

新增页面：

- 路由建议：`/geo_admin/tasks/trash`
- 显示字段：
  - 任务 ID
  - 任务名称
  - 原状态
  - 删除时间
  - 关联文章数
  - 最后运行时间
  - 操作列
- 操作：
  - 恢复
  - 永久删除

### 文章列表页

文章来源显示逻辑：

- 任务存在且未删除：显示任务名称并链接到任务编辑页。
- 任务软删除：显示“已删除任务：{任务名称}”。
- 任务已永久删除但有快照：显示“已删除任务：{source_task_name}”。
- 无任务信息：显示“无任务来源”。

### 文章编辑页

在“生成来源”模块中显示：

- 来源任务名称
- 任务状态
- 是否已删除
- 删除时间
- 如任务已删除，不显示编辑链接，只显示历史信息。
