# 服务器部署优化清单

基于 2026-06-07 系统自检结果，以下优化项在 Entity-to-Entity 关联功能完成后执行。

---

## 1. Queue Worker 超时延长

**问题：** AI 文章生成（外部 API 调用 + 多轮 RAG 检索）在长文场景可能超过 5 分钟，当前 `--timeout=300` 会被 kill。

**文件：** `docker-compose.yml` + `docker-compose.prod.yml` — queue 容器的 command

**改动：** `--timeout=300` → `--timeout=600`（dev 和 prod 同步改）

---

## 2. PostgreSQL 内存限制

**问题：** 容器内 PG 按宿主机总内存 25% 计算 shared_buffers，共享服务器上可能占用过多。

**文件：** `docker-compose.prod.yml` — postgres 容器

**改动：** 增加 command 显式设置内存参数：

```yaml
command: >
  postgres
  -c shared_buffers=256MB
  -c effective_cache_size=768MB
```

---

## 3. OPcache 配置确认

**状态：** 开发环境已有 `opcache-dev.ini`（`validate_timestamps=1`），生产环境 `opcache.ini`（`validate_timestamps=0`）。当前配置正确，无需改动。

**检查项：** 确认生产环境部署流程中不再出现 `opcache.validate_timestamps=0` 导致代码改动不可见的问题（已在 KNOWN_ISSUES.md #13 记录）。

---

## 4. pgvector 版本与向量索引监控

**问题：** `embedding_vector` 3072 维超过 pgvector 0.8.x 的 HNSW/IVFFlat 索引上限（2000 维），当前走精确检索。在 chunks 量级 < 5000 时无影响，量级增长后需关注。

**后续动作：**
- 监控 `knowledge_chunks` 表行数和 RAG 查询耗时
- 如果 chunks 破万且查询变慢时，考虑：
  - 升级 pgvector 到支持更高维度索引的版本
  - 降维策略（如 PCA 将 3072 降到 1536）
  - 或启用 `embedding_json` 的 pgvector 算子优化精确检索

**当前不需要操作。** 仅记录作为长期观察项。

---

## 5. PHP-FPM 静态资源转发（可选优化）

**状态：** 当前 Nginx 已配置静态资源 7 天缓存（`expires 7d`），无需 PHP 处理。配置正确。

**可选：** 如果未来静态资源量大，可启用 gzip_static 预压缩或 CDN 代理。

---

## 6. Redis 持久化与备份

**状态：** 生产环境 Redis 已开启 `--appendonly yes`（AOF 持久化），队列数据不丢失。配置正确，无需改动。

**检查项：** 确认备份脚本是否包含 `docker-data/prod/redis/` 目录。

---

## 7. 数据库备份方案（建议新增）

**当前状态：** 项目目录下有 `backups/` 文件夹，但未见自动化备份脚本。

**建议：** 在 scheduler 容器中增加每日 pg_dump cron 任务：

```php
// app/Console/Kernel.php — schedule 方法
$schedule->exec('pg_dump $DATABASE_URL > /var/www/html/backups/daily_$(date +%Y%m%d).sql')
         ->dailyAt('03:00')
         ->runInBackground();
```

或使用 docker-compose 外部的 crontab 调用 `docker exec geoflow-postgres-prod pg_dump ...`。
