# 用户需求理解与产品经理式转译规则

本文件是 GEOFlow 后续优化的前置协作规则。用户可以用业务语言、页面现象、截图、片段 HTML、操作感受或不完整描述来提出需求；Agent 负责先做产品经理式理解，再转成可执行的技术任务。

## 核心原则

1. 不要求用户使用技术专业术语。
2. 不把描述不完整直接当作阻塞；先根据现有页面、代码和历史阶段还原真实业务目标。
3. 先理解“用户想完成什么”，再判断“系统应该怎么改”。
4. 先保护现有功能、数据和已完成阶段，再做新增能力。
5. 只有遇到不可逆删除、数据迁移、权限安全、外部发布、费用调用，或多个业务方向都合理且无法从上下文判断时，才暂停询问用户。

## 需求转译流程

每次开发、修复或优化前，Agent 应先在内部完成以下转译：

1. 用户原话：保留需求中的页面、按钮、模块、现象和期望。
2. 产品理解：用业务语言复述“用户真正想解决的问题”。
3. 影响范围：识别涉及的页面、数据对象、生成流程、后台入口和历史功能。
4. 期望行为：明确改完后用户应该看到什么、能做什么。
5. 边界保护：判断哪些旧功能、旧数据、旧入口不能被破坏。
6. 验收标准：形成可检查的结果，例如页面显示、按钮可点、数据保存、测试通过。
7. 待确认问题：只保留真正需要用户决策的问题，不用技术细节打断用户。

## 对用户沟通方式

- 先用一句话说明“我理解你是要……”，再行动。
- 如果需要确认，用业务选项或简短问题表达，不用实现细节反问。
- 用户情绪强烈时，先拆解事实、目标和影响范围，不评价语气。
- 完成后说明改了哪里、现在应该怎么用、我做了哪些验证。
- 不用“你需要提供技术参数”作为默认回复；能从代码、页面和文档判断的，由 Agent 自行查证。

## 基本优化流程

1. 需求转译：先按本文件把自然语言需求转成产品需求。
2. 现状定位：查代码、页面、文档和历史阶段，确认现有实现。
3. 冲突检查：检查是否影响 Collection / Entity / Tag / Knowledge Base / Case / CRM 等既有边界。
4. 最小方案：优先做增量、兼容、可验证的实现。
5. 实施修改：遵守项目已有 UI、Blade、Laravel、测试和文档规范。
6. 核心自检：完成后检查功能入口、前端交互、数据保存、旧流程兼容和必要测试。
7. 文档同步：只更新受影响的 agent 文档、功能说明、更新日志和 README。
8. 交付说明：用用户能理解的语言说明结果，不只列技术文件。

## 需求转译模板

新 agent 接手复杂需求时，可以用这个模板快速落地：

```md
用户原话：

产品理解：

影响页面/流程：

期望结果：

必须保护：

验收标准：

需要用户确认的问题：
```

## 适用范围

本规则适用于：

- 新功能开发。
- 既有功能优化。
- 页面样式和交互修复。
- 生成流程、素材体系、RAG、CRM、文章编辑器等复杂模块调整。
- 换 agent、上下文压缩或隔天继续开发时的低 token 接手。
