## 1. 为什么需要重构

当前 CRM 已具备客户、询盘、商机、活动、待办和单据等对象，但它们主要是按页面分别实现的，缺少统一的销售项目链路。

典型症状：

- 客户页和询盘页都可以记录活动，但商机页没有统一活动时间线。
- 活动记录与下一步待办彼此独立，无法在记录沟通结果时顺手创建下一步。
- 询盘待办和商机待办使用同一张表，却分别只写 `inquiry_id` 或 `opportunity_id`，同一动作因此被拆成两个视图。
- 从全局“新增商机”入口创建时没有可见的来源询盘选择器，容易产生没有来源的孤立商机。
- 从询盘转商机时，没有把现有未完成待办和已有单据补充关联到商机。
- 单据同时支持 `inquiry_id` 和 `opportunity_id`，但两个选择器缺少一致性约束，可能形成错误组合。
- 商机缺少清晰的删除/归档入口，测试中容易留下重复商机。

这不是简单缺少几个按钮，而是当前系统缺少一个统一的“销售项目聚合根”。

## 3. 目标业务模型

推荐统一为：

```text
客户 Customer
  ├─ 联系人 Contact
  ├─ 客户级活动 Activity
  └─ 询盘 Inquiry：客户最初提出的需求
       └─ 商机 Opportunity：确认有采购可能后的销售项目
            ├─ 活动 Activity：该销售项目已经发生的沟通
            ├─ 待办 Task：该销售项目未来要完成的动作
            └─ 单据 Document：报价、PI、CI、装箱单、合同
```

核心原则：

1. 询盘保存来源和客户原始需求，不承担销售预测。
2. 商机保存成交阶段、金额、概率、预计成交日期和输单原因。
3. 活动保存已经发生的事实。
4. 待办保存未来动作。
5. 单据是销售项目产生的商业文件。
6. 同一条待办、活动或单据不复制，通过多个外键让不同页面查看同一条记录。
7. 一条询盘默认最多转成一个主商机，避免重复创建。

## 4. 对象职责与边界

### 4.1 客户

客户是公司/账户级对象，负责：

- 公司基础资料。
- 外部联系人。
- 内部负责人。
- 汇总该客户所有询盘、商机、活动、待办和商业单据。

客户页是跨项目总览，不应成为项目推进的唯一操作页。

### 4.2 询盘

询盘负责：

- 保存邮件、网站表单、聊天记录等原始需求。
- AI 需求识别。
- 关联 Entity、知识库、Case 和标签。
- 在尚未确认采购可能时承载初步活动和待办。
- 作为商机的来源证据。

询盘不负责：

- 金额预测。
- 赢单/输单管道。
- 商务谈判阶段。

### 4.3 商机

商机是成交推进的主记录，负责：

- 来源询盘或明确标记的“无来源直接商机”。
- 客户、主要联系人、负责人和 Collection。
- 阶段、金额、概率、预计成交时间、竞争对手和输单原因。
- 聚合该项目的活动、待办和单据。

商机不重复维护询盘中的 Entity、知识库和 Case；这些作为来源上下文只读展示。

### 4.4 活动记录

活动记录表示已经发生的事情：

- 收到或发送邮件。
- 电话、会议或在线聊天。
- 客户确认需求、价格、付款、运输或技术事项。
- 样品收发、方案反馈等事实。

一条活动应支持同时关联：

- `customer_id`：必填。
- `inquiry_id`：可空。
- `opportunity_id`：可空，新增。
- `task_id`：可空，新增，用于“完成待办并记录结果”。

客户页显示全部活动；询盘页显示该询盘活动；商机页显示该商机活动。它们是同一批数据的不同视图。

### 4.5 CRM 待办

待办表示未来必须执行的动作：

- 回复客户。
- 发送技术资料或报价。
- 等待并确认 PO。
- 跟进付款、样品、运输或审批。

一条待办可以同时保留：

- `customer_id`
- `inquiry_id`
- `opportunity_id`

询盘转商机后，不复制待办，只给未完成待办补充 `opportunity_id`。这样询盘和商机看到的是同一条记录，完成状态不会产生分叉。

### 4.6 单据

单据继续保留：

- `inquiry_id`：追溯最初需求。
- `opportunity_id`：归属销售项目。

当来源询盘已经转为商机时，新单据原则上必须关联该商机。历史单据在转商机时自动补充 `opportunity_id`，不创建副本。

## 5. 一致性规则

### 5.1 询盘与商机

- 默认一条询盘只能拥有一个未归档主商机。
- 从询盘详情点击“创建商机”时，若已有关联商机，直接进入该商机，不再创建第二条。
- 全局新增商机必须显示“来源询盘”选择器。
- 允许显式选择“无来源询盘”，只适用于复购、主动销售、老客户追加项目。
- 无来源商机必须显示“直接创建”标记，不能悄悄留下空来源。
- 已存在的无来源商机可在编辑页补关联一个同客户询盘。

### 5.2 客户和 Collection 一致性

- 选择来源询盘后，商机的客户和 Collection 自动带入并锁定或给出明确覆盖警告。
- 商机、询盘、待办和单据的客户必须一致。
- Collection 不一致时禁止静默保存。

### 5.3 待办一致性

- 从询盘创建待办：保存 `customer_id + inquiry_id`；若询盘已有商机，同时保存 `opportunity_id`。
- 从商机创建待办：保存 `customer_id + opportunity_id + source_inquiry_id`。
- 转商机时：为来源询盘全部未完成待办补充 `opportunity_id`。
- 已完成历史待办不强制迁移，可在审计报告中列出。

### 5.4 单据一致性

- 选择询盘后，如果该询盘已有商机，商机自动选中。
- 选择商机后，来源询盘自动选中。
- 如果用户选择不匹配的询盘和商机，后端必须拒绝保存。
- 转商机时，为该询盘下 `opportunity_id IS NULL` 的单据补充新商机 ID。
- 已关联其他商机的单据不得自动覆盖，必须进入冲突清单人工处理。

## 6. 询盘转商机的原子流程

新增 `OpportunityConversionService`，在数据库事务中完成：

1. 锁定询盘，防止并发重复转换。
2. 检查是否已有未归档商机。
3. 创建商机并设置 `source_inquiry_id`。
4. 带入客户、Collection、负责人、联系人和需求摘要。
5. 为该询盘未完成待办补充 `opportunity_id`。
6. 为该询盘未关联商机的单据补充 `opportunity_id`。
7. 将询盘状态更新为 `converted`。
8. 写入管理员活动日志，记录关联了多少待办和单据。
9. 返回转换结果摘要。

前端确认区应提前显示：

```text
将创建商机：桌面PCB围坝点胶
将关联：1 条未完成待办、3 份现有单据
不会复制或删除原数据
```

## 7. 活动与待办联动

二者保持独立，但提供双向快捷操作。

### 7.1 记录活动时

活动表单增加折叠项“同时创建下一步待办”：

- 待办标题。
- 截止时间。
- 优先级。
- 负责人。

勾选后在同一事务中创建活动和待办，并自动继承当前客户、询盘和商机关系。

### 7.2 完成待办时

待办提供两个动作：

- `直接完成`：仅更新状态。
- `完成并记录结果`：输入沟通结果，创建关联活动后完成待办。

不强制每次活动都创建待办，也不强制每个待办完成时都写活动，避免增加无意义操作。

## 8. 商机删除、重复与恢复

### 8.1 删除入口

商机列表卡片和商机详情页均增加删除入口，但底层使用软删除/归档。

删除确认框必须显示：

- 商机名称。
- 来源询盘。
- 未完成待办数量。
- 关联单据数量。
- 明确说明不会删除询盘、活动、待办和单据。

### 8.2 删除后的关系

- 询盘、待办、活动和单据保留原 `opportunity_id`，用于审计。
- 页面显示“关联商机已归档”。
- 来源询盘可以恢复原商机，也可以在确认后创建替代商机。
- 不使用数据库级联删除商业记录。

### 8.3 商机回收站

商机列表增加“已归档”筛选，支持：

- 恢复商机。
- 查看归档商机。
- 永久删除仅作为管理员治理功能，并且只有完全无关联记录时才允许。

### 8.4 重复商机处理

第一版不做复杂自动合并，提供“迁移关联并归档重复项”：

1. 选择保留商机。
2. 将重复商机的待办、活动和单据迁移到保留商机。
3. 校验客户和 Collection 一致。
4. 归档重复商机。
5. 写入操作日志。

## 9. 历史数据治理

由于当前历史数据较少，可以进行一次受控整理，但仍必须先预览再写入。

新增审计命令或后台报告：

```bash
php artisan crm:pipeline-audit
```

报告至少识别：

- 无来源询盘的商机。
- 一个询盘对应多个商机。
- 已有商机但待办未关联商机。
- 已有商机但单据未关联商机。
- 询盘、商机、单据客户不一致。
- 商机已归档但仍有未完成待办。
- 活动只有客户关系、但可明确归属询盘或商机的候选记录。

写入修复必须通过明确参数执行：

```bash
php artisan crm:pipeline-audit --apply
```

“桌面PCB围坝点胶”建议作为首个修复样本：

- 将商机 `#3` 关联询盘 `#7`。
- 将待办 `#1` 补充关联商机 `#3`。
- 将单据 `#17/#18/#19` 补充关联商机 `#3`。
- 不复制数据。

执行前必须再次读取实时数据，不能仅依赖本文档中的 ID。

### 阶段 6：历史数据修复与重复治理

目标：修复当前少量历史断链数据。

执行内容：

## 11. 每阶段恢复点规范

Git tag 只能恢复代码，不能恢复数据库。因此每个阶段完成后必须同时保存：

1. Git commit。
2. Git tag。
3. PostgreSQL 数据库备份。
4. 阶段测试结果。
5. 当前 `php artisan migrate:status` 输出。

建议目录：

```text
~/Desktop/GEOFlow_backups/crm-pipeline-v2/
  stage-0-baseline/
    database.dump
    git-revision.txt
    migrate-status.txt
    test-result.txt
```

阶段结束顺序：

```text
实现 -> 测试 -> UI 检查 -> 数据库备份 -> commit -> tag -> 更新阶段记录
```

任何迁移数据库的阶段，在执行迁移前额外保存一次 `pre-migration` 备份。

参考命令（执行时将阶段名替换为当前阶段）：

```bash
STAMP=$(date +%Y%m%d-%H%M)
STAGE="stage-0-baseline"
BACKUP_DIR="$HOME/Desktop/GEOFlow_backups/crm-pipeline-v2/${STAGE}-${STAMP}"
mkdir -p "$BACKUP_DIR"

docker exec geoflow-postgres sh -lc \
  'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" --format=custom' \
  > "$BACKUP_DIR/database.dump"

git rev-parse HEAD > "$BACKUP_DIR/git-revision.txt"
docker exec geoflow-app php artisan migrate:status \
  > "$BACKUP_DIR/migrate-status.txt"
```

完成测试并提交后创建代码断点：

```bash
git tag -a "crm-pipeline-v2-${STAGE}-${STAMP}" -m "CRM Pipeline V2 ${STAGE} checkpoint"
```

数据库恢复属于破坏性操作，必须先停止写入并再次确认备份文件。参考恢复命令：

```bash
docker exec -i geoflow-postgres sh -lc \
  'pg_restore --clean --if-exists -U "$POSTGRES_USER" -d "$POSTGRES_DB"' \
  < /absolute/path/to/database.dump
```

## 12. 测试矩阵

必须覆盖：

- 从询盘创建商机。
- 全局创建商机并选择来源询盘。
- 显式创建无来源商机。
- 同一询盘重复创建被阻止。
- 询盘未完成待办补关联商机且不复制。
- 商机新建待办同时保留来源询盘。
- 已有单据补关联商机且不复制。
- 单据的询盘、商机和客户不一致时拒绝保存。
- 商机活动在客户、询盘、商机三个页面一致显示。
- 活动同时创建待办。
- 完成待办并生成活动。
- 归档商机不删除单据、待办和活动。
- 恢复商机后所有关联仍然存在。
- 重复商机关联迁移后不丢数据。

## 13. UI 原则

- 商机创建页先选来源模式，再填写销售字段。
- 关联摘要使用清晰的数量和对象名称，不使用隐藏字段表达关键关系。
- 活动与待办分区相邻，但标题和说明明确区分“已发生 / 将要做”。
- 商机页展示完整来源链：客户 → 询盘 → 商机 → 单据。
- 删除按钮可见但不突出，确认对话框必须展示关联影响。
- 复用现有 CRM 导航、活动编辑器、`task-form`、`task-row` 和统一表单边框规则。
- 不新增第二套活动或待办组件。

## 14. 非目标

本轮不做：

- 完整 ERP 库存、采购、生产和财务。
- 自动发送邮件或聊天消息。
- 复杂销售自动化和审批引擎。
- 多商机自动拆分算法。
- 用 AI 自动决定历史数据关联。
- 删除询盘、待办或单据的现有表和历史字段。

## 15. 风险与决策原则

- 自动关联只能在客户、Collection、来源关系完全一致时执行。
- 有歧义的数据进入冲突清单，不猜测。
- 商机归档不得级联删除商业记录。
- 转换和历史修复必须使用数据库事务。
- 所有关联变更写入管理员操作日志。
- 每个阶段完成后先自检，再进入下一阶段。
- 若需要改变“一询盘一个主商机”的规则，必须暂停并由用户重新决策。

## 16. 推荐执行顺序

严格按以下顺序：

1. 阶段 0：冻结基线。
2. 阶段 1：数据契约和审计。
3. 阶段 2：商机生命周期。
4. 阶段 3：转换编排。
5. 阶段 4：活动时间线。
6. 阶段 5：单据一致性。
7. 阶段 6：历史修复。
8. 阶段 7：UI、测试和文档收口。

不要先做历史数据修复，也不要先在页面增加更多独立按钮；先让数据契约和转换服务成为唯一事实源。
